# write-progress

项目推进过程的写入规范。所有项目类型通用，覆盖从推进容器建立到节点追加的全流程。

## 加载链

**上游**：
- `create-project` skill — 项目入口建好后，首次需要推进记录时调用
- AI 自驱 — 向推论链追加节点 / 新建或追加推进展开子文件时调用

**下游**：
- `01 项目区/[项目]/_progress/` — 推进容器
- `01 项目区/[项目]/_progress/[管家文件].md` — 推论链（按问题逻辑链组织的节点骨架）
- `01 项目区/[项目]/_progress/[子文件].md` — 推进展开（具体论证 / 证据 / 分支）

**同级联动**：
- `new-file/SKILL.md` — 新建管家文件或推进展开子文件时，强制补加载链

---

## 角色与文件关系

| 角色 | 文件 | 写什么 |
|---|---|---|
| 项目入口 | `_overview.md` | 项目顶层状态 / 文件清单 / 相关项目（非本 skill 管辖） |
| 推论链 | `_progress/[管家文件].md` | 按问题逻辑链组织的节点骨架 + 指针 |
| 推进展开 | `_progress/[子文件].md` | 节点背后的具体推论 / 证据 / 分支 |

**核心理念**：推进记录的本质是"从 A 到 B 是怎么走过来的"——因果逻辑链，不是时序流水账。逻辑承接是组织轴；日期作为字段标注，不作为组织维度。
### v5 · 记忆边界

`_progress` / In_Progress 是项目推论链权威源，记录问题发现、转向、开出与收束。

它不是记忆系统写入落点：

- 不直接写 `episodic_inbox.md`。
- 不直接写 `episodic_memory.md`。
- 不直接写 `semantic_memory.md`。
- 不替代 MEMORY_LOG。

后续 close-node / weekly-review 可以从 `_progress` 中提取两类东西：

1. **转向理由**：为什么从问题 A 走到问题 B。
2. **可迁移 schema**：如果跨情景成立，再由 close-node / daily-review / weekly-review 按记忆区规则进入情景层或语义层。

因此，本 skill 只负责把项目推论链写清楚；记忆提取与代谢不在本 skill 内执行。

### 1.2 节点格式

每个节点固定字段：

```markdown
### [字母编号]. [问题/命题简短命名]（YYYY-MM-DD）
- **起点**：上游遗留问题或新出现问题（一句话）
- **推进**：调研 / 讨论 / 推论的简短摘要（2-4 行，不展开论证）
- **结论**：本节点收敛出什么（一句话；未收敛写"未解决 / 悬置"）
- **新问题**：→ [下一节点编号]（一句话描述开出的新问题）
- **详记**：→ `具体文件名.md` 或 `具体文件名.md §对应节`
```

**字段约束**：

- **编号**：用字母（A/B/C/...）按推进顺序追加；分支用并列编号（B 解决 → C；B 未解决 → D 并存）
- **日期**：节点首次写入日期；节点内容后续更新可在"推进"或"结论"字段追加日期标注
- **摘要简洁不严谨**：只写"发现什么 / 要做什么 / 走到哪了"，不复述论证
- **段间转换标记**：`推进` 字段内每一步若产生下一步，必须写 `**开出**：[下一步问题/方向]`。此标记用于记录项目推论链中的转向理由，供 close-node / weekly-review 事后提取；它本身不是记忆写入动作。
- **新问题必填**：节点末尾固定保留 `新问题` 字段；没有开放问题时写"无开放问题"，不可留空
- **指针粒度灵活**：可以指整个文件，也可以指文件某节——按内容实际分布写

### 1.3 子问题嵌套与分支

**子问题嵌套**：一个大节点展开出 B1/B2/B3 等子问题，**不在推论链单独成节**——下沉到 B 的详记子文件内部。推论链只到 B 这一层。

```markdown
### B. [大问题命名]（YYYY-MM-DD）
- 起点：承接 A 开出的"..."
- 推进：拆出子问题 B1/B2/B3 分别推进
- 结论：B1 解决，B2 待解，B3 引出分支
- 新问题：→ C（承接 B1）/ → D（承接 B3）
- 详记：→ `XX.md §三`（B1-B3 在此展开）

### C. [承接 B1]（YYYY-MM-DD）
...

### D. [承接 B3 分支]（YYYY-MM-DD）
...
```

**分支不强行收束**：B 解决和 B 未解决可分别开 C 和 D 并存。若后续再交汇，新建一个节点说明承接哪条线。

## Part 2 · 推进展开（子文件）

### 2.1 首次建立

- 文件名按主题命名（中文可），不带日期
- 顶部 frontmatter + 加载链（上游 = 推论链管家文件）
- 同步更新推论链对应节点的"详记"字段

### 2.2 组织原则

**按"问题推进单元"组织**，每节是一个推进单元：

1. **上步问题**（本节起点）
2. **推论过程**（证据 / 论证步骤 / 分支判断 / 调研内容）
3. **结论**（本节收敛出的认识）
4. **新问题**（引出下一节起点）

**节标题按"问题或结论"命名**，不按日期：
- 好：`§三 为什么 X 不能归 Y`
- 差：`§三 04-15 讨论记录`

**日期的用法**：作为节内字段标注（"04-15 讨论得出..."）帮助梳理逻辑演进时间线，不作为节的组织轴。

**调研内容附属在触发问题下**：文献调研、概念辨析写在引发它的问题节内，不独立成节。

### 2.3 追加与扩展

- **同节继续展开**：按推论链在原节内追加，不另起时序流水节
- **开出新问题**：节末写 `> **→ 开出的新问题**：...`；若本节闭合且没有新问题，写 `> **→ 本节收束**：...`
- **新主题独立**：若新问题足以独立成主题，新建一个推进展开子文件，不无限延长当前文件

### 2.4 推论链导览 + 节字段（强制格式）

每个推进展开子文件必须包含两套结构：**顶部推论链导览** + **每节问题承接/开出字段**。

#### 推论链导览（顶部）

紧接在加载链 + 引言指针块之后、第一个内容节之前。**作用是用一段叙事把全文节点串成一条可独立读懂的推理路径**，读者只看导览就能复现"从入口问题到当前结论"的完整推导，不必钻进每节内文。

**核心原则**：导览不是节点目录。每个节点描述必须**包含"为什么从上一节走到本节"的桥句**——上一节做完后留下了什么缺口、为什么这个缺口必须用本节来补、本节的什么能力正好能补这个缺口。只贴"§X 是 Y"的标签型描述是失败的导览。

**模板**：

```markdown
## 推论链导览

入口问题：**[本文件要回答的核心问题]**

[链型说明：问题推导链 / 证据汇聚 / 反例对照 / 等等。一句话点明本文件属于哪一种推论结构]

[主推导链编号列表，每条带桥句]：

1. **§一 [节标题简称]** — [本节做什么、得到什么]。**[但这留下什么缺口 / 为什么必须接下一节]**，于是 →
2. **§二 [节标题简称]** — [本节怎么补上一节的缺口、得到什么]。**[新缺口 / 为什么必须再接下一节]**，于是 →
3. ...

[未完全闭合的最后缝隙 / 走到下一文件的出口，单独一行说明]
```

**对比示意**：

| | 失败（节点目录） | 成功（推导链） |
|---|---|---|
| 写法 | "§2 Neubauer — 专家神经效率更高" | "§2 Neubauer — §1 只证明协同重要，但没排除算力扩容并存的可能；神经效率假说提供专家发展轨迹观测：算力反向下降，扩容读法被排除。但'协同'还停在抽象层，没锚到具体环路 →" |
| 区别 | 只贴"是什么"标签 | 含"为什么从上一节走到本节"的桥句 |

**引入时机**：
- 子文件超过 3 节 → 必须有导览
- 1-3 节阶段 → 导览可省，节字段必加
- 新节追加时 → 同步更新导览相应条目

#### 节字段（每节强制）

每节标题下用 blockquote 加两个字段，包夹节内原文：

```markdown
## [节标题]

> **问题承接**：[一两句话，承接上一节的开出问题 / 或承接外部前置（其他文件 §X）]

[节内推论展开 — 原有内容]

> **→ 开出的新问题**：[一句话，引向下一节 或 下一文件]
```

若本节已闭合且没有开放问题，末尾改用：

```markdown
> **→ 本节收束**：[一句话说明本节闭合了什么；没有自然下游时明确写"无开放问题"]
```

**字段约束**：
- **承接**字段必须明确指向具体的上游节或前置（不写"承接前文"这种模糊话）
- **开出**字段必须明确指向下游节或下游文件（"留至 `XX.md §Y` 回答"）；没有下游则用"本节收束"，不可省略字段
- **节内文字保持原样**——字段只在外围标注承接/开出关系，不重写节内推论

#### 整理已有文件

对已有但缺失上述格式的子文件，作为独立梳理任务执行：
1. 通读全文，识别节与节之间的实际推导关系
2. 顶部加推论链导览（含桥句，不能是节点目录）
3. 每节两端加承接/开出字段
4. **节内文字一字不动**——这是历史推动记录，梳理是为既有内容补逻辑骨架视图，不是改内容

## 推论链 / 推进展开 边界

| 层 | 写什么 | 不写什么 |
|---|---|---|
| 推论链 | 节点骨架 + 简短摘要 + 指针 | 论证展开 / 证据细节 / 分支讨论 |
| 推进展开 | 推论展开 / 证据 / 分支 / 失败路径 | 跨主题的全局推进顺序 |

**核心约束**：推论链看完知道"走到哪、为什么走到这"；推进展开看完知道"具体怎么推出来的"。
### 与记忆系统的边界

| 对象 | 权威源 | 是否由本 skill 写入记忆 |
|---|---|---|
| 项目问题发现 / 转向 / 开出 | `_progress` / In_Progress | 否 |
| 项目结论 | 项目主文档 / progress 结论字段 | 否 |
| 高信号具体事件 | `episodic_inbox.md` / `episodic_memory.md`，由 close-node / daily-review / weekly-review 分流 | 否 |
| 跨情景稳定 schema | `semantic_memory.md` 或毕业候选，由 weekly-review 按记忆区规则提取 | 否 |

本 skill 的责任是把推论链写到可被提取：承接清楚、开出清楚、收束清楚。是否进入记忆系统由相邻 skill 按 v5 规则判断。

## 操作流程摘要

**场景 A · 从零建推论链**（上游：create-project 或用户首次推进请求）：
1. 读项目 `_overview.md` 获取 `project_type`
2. 建 `_progress/` 目录
3. 调用 `new-file` skill 建管家文件加载链，再写入管家骨架（按类型命名）
4. 按节点格式写入首节点
5. 更新 `_overview.md §管辖文件 / §项目文件`

**场景 B · 追加推论链节点**：
1. 读管家文件末节的"新问题"字段确认承接
2. 新节点编号承接（A→B→C；分支用并列编号）
3. 按节点格式填入起点 / 推进 / 结论 / 新问题 / 详记
4. `推进` 内若产生下一步，写 `**开出**`；`新问题` 无开放项也要写"无开放问题"
5. 若引出新推进展开子文件，同步建子文件并按问题推进单元起始

**场景 C · 新建推进展开子文件**：
1. 文件名按主题命名
2. 调用 `new-file` skill 写顶部 frontmatter + 加载链
3. 首节按问题推进单元起头，并写 `问题承接`
4. 节末写 `→ 开出的新问题` 或 `→ 本节收束`
5. 更新推论链节点的"详记"字段指针

**场景 D · 追加推进展开现有节**：
1. 在节内按推论链展开
2. 新增二级节时，节首写 `问题承接`
3. 节末写 `→ 开出的新问题` 承接下一节/新子文件；若闭合，写 `→ 本节收束`

## 注意

- **组织轴是问题逻辑链，日期只是字段标注**——禁止时序流水节标题
- **推论链不写论证**——展开全部下沉到子文件
- **段间转换不可省略**——L2 用 `**开出**` / `新问题`，L3 用 `问题承接` + `→ 开出的新问题` / `→ 本节收束`
- **指针粒度灵活**——文件 / 文件§节都可
- **分支保留并行**——不强行合并未交汇的分支
- **文献规范不在本 skill**——见 `manage-research-reference`（待建）
- **不直接写记忆池**——`_progress` 是项目推论链权威源，不是 `episodic_inbox.md`、`episodic_memory.md` 或 `semantic_memory.md` 的写入落点。
- **不把段间转换自动抽象为 S**——只有 close-node / weekly-review 在读取转向理由后，才能判断是否形成可迁移 schema。
- **不替代记忆代谢**——升星、衰减、毕业、情景层归档均不在本 skill 执行。

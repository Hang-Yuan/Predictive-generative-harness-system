# close-node

## 加载链（上下游）

- **上游**：AI 自判触发 / daily-review 节点闭合兜底 / 用户明确要求节点收束。
- **管辖文件（下游）**：
  - 当前项目主文档 / `_overview.md` / `_progress/` — 节点闭合的项目权威落点。
  - `<ASSISTANT_ROOT>\MEMORY\episodic_inbox.md` — 节点级实时校准信号。
  - `<ASSISTANT_ROOT>\MEMORY\episodic_memory.md` — 节点内已复现的情景级候选。
- **同级联动**：
  - `write-progress/SKILL.md` — close-node 步骤 4 调用 write-progress，且不把 progress 当记忆池。
  - `daily-review/SKILL.md` — 日级兜底检查漏 close 的节点。
  - `weekly-review/SKILL.md` — 周级 semantic 升降、毕业候选与情景层衰减。
  - `new-file/SKILL.md` — 新建文件时强制补加载链。

---

## 定位

close-node 是节点闭合时的多文件同步流程。它比 daily-review 粒度更细，负责在一个讨论 / 工作节点收束时及时写入项目文件，并对该节点中出现的记忆信号做分流。

v5 修正点：

- In_Progress / `_progress` 是项目推论链权威源，不是记忆写入落点。
- 默认动作是遗忘；close-node 只处理当前节点内已经清楚的校准信号。
- hook 已做实时两轴标注；close-node 的角色是补齐漏记、聚合同类、合并证据，而不是重新判全量信号。
- 节点级只写 `episodic_inbox.md` 或 `episodic_memory.md`；不写 `semantic_memory.md`。
- close-node 不依赖 hook，不改 hook。

---

## 触发

AI 自判触发。识别信号包括：

- 一段连续讨论 / 工作收束。
- 子问题解决。
- 决策拍板。
- 具体任务完成。
- 用户说「好了 / 收 / 下一个 / 这块告一段落」等显式信号。

触发时 N 级提议：

> 这段按节点闭合处理？涉及 [项目X]。

得到确认后执行。用户已经明确授权当前项目闭合时，可直接执行。

---

## 流程

### 步骤 1 · 项目定位

判断本节点归属哪个项目：

- 清楚 → 继续。
- 模糊 → 向用户确认。
- 跨项目 → 选主要权威落点，其他项目只留指针；跨项目影响按 C 级确认。

---

### 步骤 2 · 主文档（强制判断 + C 级 verdict）

**硬约束**：每次 close-node 必须显式输出主文档判断，不允许默认跳过。

输出格式（呈给用户）：

```
主文档判断：[需要更新 / 不需要更新]
若需要：拟更新节 = §X，拟加内容 = ...（具体到节级 + 内容要点）
若不需要：理由 = ...（必须给出，不能写"无"或"协议层执行"等空泛词）
```

**执行规则**：

- "需要更新" → 等用户 C verdict 后才动主文档；AI自己写（研究性主文档不外包给检索 sub-agent）
- "不需要更新" → 等用户 C verdict 后才进入步骤 3；用户可纠正"其实需要"
- 输出后直接静默等 verdict，不假设默认值

**不允许的判断模式**（标准滑动陷阱）：

- "本节点是协议层执行 / 工程动作，不是论证更新" — 滑动：把"是否提供论据"换成"是否变更顶层架构"
- "主文档当前没人在写 / 本节点没产生新结论" — 滑动：把"主文档是否需要追"换成"主文档当前状态"
- "_overview / progress 已经覆盖" — 滑动：_overview 是状态指针、progress 是推论链，二者均不替代主文档；主文档承担论证 / 案例 / 理论锚点扩展

**判准（不分项目类型，统一用）**：

本节点是否提供以下任一类内容 → 需要更新：

- 新论据 / 新案例 / 新数据点（验证或反驳主文档已有论点）
- 理论锚点扩展（主文档引用的理论框架被本节点延展 / 修正）
- 新文献支撑（应进 §References 或对应理论节）
- 决策固化（影响主文档已有结论 / 边界 / 设计原则）
- 框架更新（顶层架构 / 关键定义变更）

任何一项符合 → "需要更新"。全不符合 → "不需要更新" + 给具体理由。

---

### 步骤 3 · `_overview.md`

按项目结构更新：

- §当前状态：推进到哪、断点在哪。
- §待解问题 / §悬置：关闭已解决项，追加新涌现项。
- §架构演进表：如有则旧→新追加。
- §管辖文件：如新增文件则补指针。

---

### 步骤 4 · `_progress/`

调用 `write-progress`：

- 追加推进节点，或更新现有节点字段。
- 确保 L2 / L3 段间转换标记齐全。

检查项：

- L2 `推进` 中产生下一步的地方，有 `**开出**`。
- L2 节点末尾有 `新问题`，无开放项写「无开放问题」。
- L3 新增 / 更新节首有 `> **问题承接**`。
- L3 节末有 `> **→ 开出的新问题**` 或 `> **→ 本节收束**`。

这些标记供 close-node / weekly-review 事后提取转向理由，不代表直接写入记忆池。

---

### 步骤 4.5 · 节点闭合后的记忆分流

输入限本次闭合节点涉及的内容，不全项目扫：

- 本轮 touched `_progress/` 段间转换标记。
- 本节点对话中出现的高信号具体事件。
- 本节点形成的可迁移校准信号。

#### 4.5a · 身份层前置过滤

先匹配 USER / SOUL / 已启动注入的 semantic schema：

| 结果 | 动作 |
|---|---|
| 命中身份层 + 符合旧认知 | 不写 |
| 命中身份层 + 出现抵触 / 新扰动 | 写 `episodic_inbox.md` 或提交裁决 |
| 未命中身份层 | 继续分流 |

#### 4.5b · 两轴字段复用 / 补齐

hook 已经写入的条目，复用其 P轴 / C轴字段。close-node 只在两种情况下补标：

1. 节点内有明确校准信号，但 hook 未写入。
2. 多条 inbox 信号需要合并成一个情景级候选。

补标字段：

| 轴 | 值 |
|---|---|
| P轴 | 命中 / 击穿 / 未覆盖 |
| C轴 | 正向确认 / 反对 / 纠正 / 中性 |

节点级写入下限：

```text
P轴=命中 且 C轴=中性 -> 不写
其他组合 -> 补写 inbox 或聚合进 episodic_memory
```

#### 4.5c · 分流判准

| 类型 | 落点 |
|---|---|
| 项目结论 / 具体内容 | 主文档 / progress，不进记忆池 |
| 问题发现与转向理由 | progress 保持权威源；必要时 weekly-review 从中提取 |
| 单次校准信号 | `episodic_inbox.md` |
| 同一节点内多次复现且边界清楚 | `episodic_memory.md` |
| 可迁移但证据不足 | `episodic_inbox.md`，等待 daily-review / weekly-review |
| 已可抽象为语义 schema | close-node 只标记，不直接写 `semantic_memory.md` |

#### 4.5d · 写入约束

- `episodic_inbox.md` 只允许追加到活动表末尾；新行必须严格 9 列、首尾有 `|`、不得含额外 `|`、不得插入空行或重建表头；编号按同一物理 inbox 内当日最大序号取下一个未占用值。L0 行级写入 / 删除均为 N 级，执行后回报，不问 C。
- `episodic_memory.md` 只写 1-3 星情景级候选，不启动注入。
- `semantic_memory.md` 只由 weekly-review 升降。
- 项目结论仍留在项目权威源。
- 单纯新增 inbox 条目不写 `MEMORY_LOG`；升格、批量衰减、归档时再写。被升格 / 吸收 / 修正 / 无保留价值的 inbox 行默认物理删除，不回写状态堆积历史；删除整份文件或非 inbox 历史证据仍为 C 级。

---

### 步骤 5 · 加载链 / 指针

扫描受影响文件：

- frontmatter。
- `## 加载链（上下游）`。
- 跨文件指针。

动作：

- 新建文件 → 调用 `new-file` 规则补加载链。
- 既有指针失效 → 修正。
- 新关联关系 → 补同级联动。

---

### 步骤 6 · `_本周.md §本周任务`

最小任务维护：

- 本节点覆盖的 checkbox → 勾为 `[x]`。
- 新涌现任务 → 追加 `[ ]`。
- 过时描述 → 改写或标注。

---

### 步骤 7 · 悬置与交接

- 悬置点 / 未解议题 / 待续讨论 → 写 `_本周.md` 或 `长期记忆.md §当前处境 §上次未决`。
- 节点记忆信号已由步骤 4.5 分流，不再交给 hook。
- close-node 只做节点级提取和指针，不做候选池代谢、衰减、毕业。

## 不做

- 不把 progress 当记忆池。
- 不使用项目推进记录到语义层的直通写入。
- 不写旧程序 / 语义双池。
- 不写 `semantic_memory.md`。
- 不做候选池代谢、衰减、毕业。
- 不触发身份层修改。
- 不动 hook / settings。

---

## 执行等级

- 触发提议：N。
- 项目内常规同步（_overview / progress / _本周）：S/N，按项目权限与既有规则。
- **步骤 2 主文档判断输出本身**：N（必须输出，等 verdict）。
- **步骤 2 写主文档**：C（无论"需要更新"还是用户授权"不更新"，都要 verdict 后动）。
- 跨项目影响 / 覆盖既有主文档结论 / 身份层相关：C。
- 本地删除或物理删除整份记忆文件：C。L0 `episodic_inbox.md` 行级写入 / 删除：N。

---

## 不允许外包

研究性内容（论证 / 理论锚点 / 案例分析 / 文献整合）由AI独立撰写，不派 sub-agent。检索 sub-agent 可承担：文件归档 / 命名规整 / 多用户重组 / public repo 推送 / 代码层调试 / 文档草拟（事务性）。研究性主文档撰写跨过外包边界——这条这条边界确保身份层产出由 AI 主体直接负责。

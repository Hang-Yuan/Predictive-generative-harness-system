# close-node

节点级闭合 skill。把 daily-review 里的"全日扫描"在节点粒度上提前做一次，避免长工作日（10+ 小时）到复盘时前段细节丢失。

## 加载链（上下游）

- **上游**：AI 自判触发 · CLAUDE.md §B 触发表
- **同级联动**：
  - `write-progress/SKILL.md` — 步骤 4 调用
  - `new-file/SKILL.md` — 新建文件时强制补加载链
  - `daily-review/SKILL.md` — 日级复盘，里面加兜底"今日节点是否都 close 过"

---

## 触发

**AI 自判**（不靠用户口述闭合信号，用户多数时候不会主动标）。

识别信号（非穷举）：

- 一段连续讨论 / 工作收束：用户语气转换、主动说"好了/收/下一个"、话题切换
- 一个子问题被解决：结论达成、决策拍板、议题闭合
- 一个具体任务完成：实现落盘、主文档写入、bug 修复
- 用户口述"这块收了 / 告一段落 / 先到这"等显式信号

触发时 N 级提议：「这段按节点闭合处理？涉及 [项目X]。」
得确认后执行下列流程；否则挂起，不动。

---

## 流程

### 步骤 1 · 项目定位（唯一核心判断点）

判断本节点归属哪个项目。

- 清楚 → 下一步
- 模糊 → 向用户确认，不自己猜
- 跨项目 → 选主要落点项目，其他项目只留指针（C 级确认跨项目影响）

项目定位后，后续步骤全部流程化。

### 步骤 2 · 主文档

本节点是否产生需要固化到项目主文档的结论 / 决策 / 框架更新？

- 有 → 写入主文档对应节（N 级告知落点）
- 无（只是推进过程，无定型结论）→ 跳过

### 步骤 3 · `_overview.md`

- §当前状态：更新"推进到哪、断点在哪"
- §待解问题 / §悬置：关闭已解决项、追加新涌现项
- §架构演进表（如有）：按旧→新追加本次变更

### 步骤 4 · `_progress/`

调用 `write-progress` skill：追加推进节点 or 更新现有节点字段。本 skill 不重写 write-progress 规则，复用其权威源。

调用后检查段间转换标记是否齐全：
- L2 节点 `推进` 中产生下一步的地方，必须有 `**开出**`
- L2 节点末尾必须有 `新问题`；无开放项写"无开放问题"
- L3 新增/更新二级节时，节首必须有 `> **问题承接**`
- L3 节末必须有 `> **→ 开出的新问题**` 或 `> **→ 本节收束**`

### 步骤 4.5 · progress → S 候选提取（本节点级）

输入限本轮 touched 的 `_progress/` 节段间转换标记：L2 `推进` 中的 `**开出**` / L2 `新问题` / L3 `> **问题承接**` / L3 `> **→ 开出的新问题**` / L3 `> **→ 本节收束**`。不全项目扫，只看本次闭合节点涉及的段。

**判准**（粒度筛子）：

| 类型 | 处理 |
|---|---|
| 思维路径 / 方法论 / 行为模式（一年后仍成立） | → S 候选 |
| 项目结论 / 具体内容（依附本项目） | 不入池，已在步骤 2 入主文档 |
| 一次性观察 / 不可压缩到 schema | 不入池，留在 progress 原地 |

**操作**（按 §M.2 三处同步执行）：
- 命中现有 S 条目 → 主池升星 + archive 追加证据 + MEMORY_LOG 流水
- 未命中且达 schema 标准 → 主池新建 ★ + archive 写详记 + MEMORY_LOG 流水
- 未达 schema 但值得周级复审 → 暂存到 MEMORY_LOG §周复盘暂存（供 weekly-review 步骤 3.5 拉取）

S 级静默执行；提取产物简短告知用户即可，不等确认。

### 步骤 5 · 加载链 / 指针

扫描受影响文件的：
- frontmatter / §加载链 节（上游 / 管辖文件 / 同级联动）
- 跨文件指针引用（`§节名`、相对路径）

动作：
- 新建文件 → 调用 `new-file` skill 补加载链 frontmatter
- 既有指针失效（文件改名 / 节删除 / 路径迁移） → 修正
- 新关联关系 → 补 同级联动 条目

### 步骤 6 · `_本周.md §本周任务`

最小任务维护规则（直写，权威源 = 本步骤 + `CLAUDE.md §R 行为路由` 中"本周流水"路由）：
- 本节点覆盖的 checkbox → 勾为 `[x]`
- 新涌现任务 → 追加 `[ ]` 到对应项目分组末
- 过时描述 → 改写或标注

### 步骤 7 · 记忆 / 悬置

- E 类结论已在步骤 2 入主文档 → 不重复
- 本节点中涌现的 P / S 信号分两路：**progress 段间转换的 schema 提取**由步骤 4.5 处理（节点级即时入池）；**非 progress 的对话级信号**走 hook（`memory_signal.sh`）+ daily-review 兜底，本 skill 不介入
- 悬置点 / 未解议题 / 待续讨论 → 写入 `_本周.md` 或 `长期记忆.md §当前处境 §上次未决`

## 与相邻 skill 边界

| Skill | 粒度 | 职能 |
|---|---|---|
| write-progress | 推进节点级 | 写推进过程。本 skill 步骤 4 调用 |
| **close-node**（本） | 节点闭合级 | 节点闭合时的多文件协同更新 |
| daily-review | 日级 | 全日回顾 + 情绪 + 明日断点。末尾加"今日节点是否都 close 过"兜底 |
| weekly-review | 周级 | 候选池代谢 + 毕业判定 |

---

## 不做

- 不生成日级 / 周级汇总（daily/weekly review 的事）
- 不触发身份层修改（走 `CLAUDE.md §M.1`）
- 不做候选池代谢升降星（weekly-review 的事）
- 不替代 write-progress（复用它，不重写）

---

## 执行等级

- 触发提议：N（告知 + 等确认）
- 各步骤内写入：S（静默执行）
- 跨项目影响 / 覆盖既有主文档结论：C（确认）

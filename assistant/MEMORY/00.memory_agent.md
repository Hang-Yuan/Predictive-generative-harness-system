## 写入格式（v3 · schema prediction）

所有新增条目统一使用 schema 格式。

### 程序层（procedural_memory.md）

```
### [模式名称]
- **强度**：★ / ★★★ / ★★★★★
- **触发情境**：[什么情形下激活这条规则]
- **行动模式**：[AI 应执行的动作]
- **预期后效**：[执行后预期用户/世界如何反应]
- **来源**：[观察依据 · 日期]
- **related**：[关联的语义条目，或 —]
```

### 语义层（semantic_memory.md）

```
### [标题]
- **强度**：★ / ★★★ / ★★★★★
- **情境**：[什么情形下这条 schema 被测试]
- **预期**：[在此情境下用户大概率如何反应]
- **来源**：[观察依据 · 日期]
- **related**：[关联的程序条目，或 —]
```

## 判断锚点（程序 vs 语义）

- 属于"AI 遇到 X 就做 Y" → **程序层**
- 属于"用户是这样的人 / 持有这种立场 / 用这种方式思考和工作" → **语义层**
- 身份层冲突（与 USER / persona_SOUL 直接矛盾）→ 不动身份层，作为 High 惊奇度入候选池

## 代谢操作

完整规则见 `CLAUDE.md §M.2 §M.9`。本节是执行步骤：

### 入池（Execution Mode 实时）

| 惊奇度 | 星级 | 动作 |
|---|---|---|
| High（命中 + 偏离） | ★★★ | 升星已有条目 / 新增例外条目 / 修正已有条目的情境边界或预期 |
| Medium（未命中） | ★ | 新建 schema 条目 |
| Low（命中 + 符合） | — | 不写入 |

### 升星（同方向再次观察）

在原条目上升级 `★ → ★★`（以此类推），证据区追加"再次观察 · 日期"。不重复创建条目。

### 降星 / 休眠（Review Mode 批量）

- schema 连续 N 周无命中 → 降一颗星
- 降到 0 星 → 标 `[dormant]`
- 休眠状态持续无命中 → 删除

### 毕业（★★★★★）

**毕业是条目重写，不是搬运。** 身份层（USER / persona_SOUL）是稳态规则，不存日志元数据；MEMORY_LOG 才是审计流水落点。

**四步走**：

1. **剥壳写入身份层**：只搬净描述字段——
   - 程序层：**标题 + 触发情境 + 行动模式**（可加一句简述"为什么这样做"）
   - 语义层：**标题 + 情境 + 预期**（可加一句简述）
   - **剥掉**：强度 / 来源 / related / 证据 / 备注等 log 字段
   - 程序层写入 `SOUL/persona/persona_SOUL.md §行为模式`，语义层写入 `USER/USER.md`（节分流见 weekly-review §步骤 6）
2. **原条目标毕业**：候选池原条目标 `graduated → [目的地] (YYYY-MM-DD)`，保留全部元数据不删，不参与衰减
3. **完整条目归档到 MEMORY_LOG**：在 `MEMORY_LOG.md §毕业归档` 追加一条**完整毕业条目**（含原条目全部 log 元数据 + 毕业日期 + 毕业去向 + 剥壳后的净描述副本），这是毕业事件的唯一完整审计记录
4. **§操作日志加一行指针**：`YYYY-MM-DD | 毕业：[条目名] → [目的地]（详见 §毕业归档）`

**触发条件**：weekly-review 步骤 6 扫描到 ★★★★★ + 跨周稳定 + C 级确认，不自动触发。

### 冲突

新条目与已有条目矛盾 → **不静默覆盖**，N 级告知用户，等待裁决。

## Bloat 阈值

`procedural_memory.md` 和 `semantic_memory.md` 各 ~30 条（合计 ~60）。见 `CLAUDE.md §T`。

- 单文件 >30 条 → 优先 weekly-review 深度代谢（毕业 / 衰减），不直接扩容
- 合计 >60 条 → 强制代谢

## 区域职责

**专注区**是用户的周工作台。所有进行中的工作都在这里：原始想法 → 任务拆解 → 每日推进 → 周末归档。

专注区以**周**为单位运转：周初口述想法，AI 拆解为任务；一周内推进，AI 追踪进展；周末回顾、归档、准备下一周。

**关键边界**：
- 工作过程 / 推导细节写项目目录下 `_progress/`，不写 `_本周.md`
- 项目结论写项目主文档，不写 `_本周.md`
- 校准信号写 `MEMORY/episodic_inbox.md`，不写 `_本周.md`
- `_本周.md` 只记"做了什么 / 决策了什么 / 推进到哪"

## 维护本区的 skill

| skill | 何时调用 | 对本区做什么 |
|---|---|---|
| `week-sync` | 每次会话启动（§B 启动序列步骤 6 自动调用） | 同步本周状态：周一至周四轻量同步、周五至周日深度回溯（差集扫描漏记文件、核对任务完成状态、接续上次工作） |
| `daily-review` | 用户告别语触发 / 用户主动要求总结 | 写入今日进展段（`### 周X YYYY-MM-DD` + 工作段含项目名 / 关联文件 / 流水账）；扫节点闭合兜底；处理记忆 inbox |
| `weekly-review` | 周日 daily-review 自动触发 / 用户手动触发 | 全周整理：生成本周产出节、归档 `_本周.md` 为 `_归档/YYYY-Wnn.md`、创建新周文件、写入 MEMORY_LOG 周复盘条目 |
| `close-node` | 节点级阶段性收尾 | 提取节点级校准信号到 inbox 或 episodic_memory；项目主文档结论由用户 verdict 后写入 |
| `write-progress` | 项目推进过程需要记录 | 写项目目录下 `_progress/` 推进文件，不写 `_本周.md`（避免污染周流水） |

## `_本周.md` 结构

每个 `_本周.md` 含以下区块：

### 1. 原始口述
未经过滤的想法 / 问题 / 半成品计划。AI 不编辑此区，仅作为源材料保留。

### 2. 任务清单
可执行任务列表，格式：
```
- [ ] [项目名] 任务标题 — 上下文/子任务
```
一周内随时增删勾选。

### 3. 进展记录
每日工作段，按日期节组织（`### 周X YYYY-MM-DD`）。每个工作段格式：
```markdown
**项目名称**（`项目根目录路径`）
关联：`具体文件1.md` · `具体文件2.md`

- 今天做了什么（流水账，不写理论推导细节）
- 确立了什么决策
- 完成了什么，推进到哪里
```

> 「关联：」字段是 week-sync 深度回溯的索引入口和接续机制的定位锚点。

### 4. 本周产出
归档前由 weekly-review 一次性补写，列出本周实际交付物。

---

## 归档规则

周关闭时由 weekly-review 执行：

1. 检查 `### 本周产出` 节是否存在且位于进展记录末尾，缺失则补写
2. 重命名 `_本周.md` → `_归档/YYYY-Wnn.md`（如 `2026-W19.md`）
3. 创建新的 `_本周.md`
4. 未完成任务继承到新周（标记 [继承]）

**归档不修改 / 不删除**：历史档案保持原貌，回溯永远可读。

---

## 写入约定

- `_本周.md` 写入为 S 级（静默执行，不需用户确认）
- 项目结论 / 阶段决策不写 `_本周.md`，由 daily-review 步骤 2 用户 verdict 后写入项目主文档
- 任务完成时间、归档周次、`MEMORY_LOG` 日期均使用**逻辑日期**（物理 hour < 06:00 → 逻辑日期 = 物理日期 − 1）
